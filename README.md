# sfuzz

Simple blackbox fuzzer to harden the validation layer of JSON API endpoints. 

## Features

Compared to other black box API fuzzers this one was designed to have a new particular 
set of features. 

- simple fuzz file format to capture all fuzzing requests at once
- autogeneration of fuzz files from any Open API specification (> 3.x)
- converge quicker via embedded fuzz types and generation of happy path values

Importantly, as a black box fuzzer it does not mutate based on static/dynamic source code feedback.

## Resilient APIs

APIs are a front to many products and businesses. Making their validation layer
handles exotic/unexpected values yield:

- more 4xx business responses and less 5xx status codes
- less noise in logs and alerting 
- less out of bonds and overflows errors
- continuous stress testing of the business validation layer

## LLM Friendly

`sfuzz` is designed so that LLMs can also discover shortcomings in APIs:

- concise, structured and meaningful console outputs
- clear text and no colored console output
- separate steps for fuzz file generation and actual fuzzing

## How does it work?

#### Fuzz Keywords

A fuzz keyword is a placeholder which encode type and example information.

| Fuzz Keyword | Type (i.e. format)   |
|--------------|----------------------|
| FUZZSTR      | Generic string       |
| FUZZNUM      | Numeral              |
| FUZZUID      | Universal identifier |
| FUZZDTE      | Date format          |
| FUZZTME      | Time or stamp format |
| FUZZJWT      | JWT auth token       |

A fuzz keyword can embed a happy path value, i.e. an example value.

| Keyword with embedded values | Example value |
|------------------------------|---------------|
| FUZZjohnSTR                  | john          |
| FUZZ1234,34NUM               | 123,34        |
| FUZZav34ef-56cafUID          | av34ef-56caf  |
| FUZZ2024-03-05DTE            | 2024-03-05    |
| FUZZ12:43TME                 | 12:43         |

Examples serve two purposes:

1. replace keyword with a happy path value when needed
2. use example value as a starting point to mutate and fuzz with exotic values 

> [!TIP]
> Fuzz types and embedded examples are autogenerated when using Open API spec.

#### Fuzz File

A fuzz file contains fuzz requests on each line.

```txt
GET https://nice.fr/FUZZUID?name=FUZZSTR&stamp=FUZZTME
...
POST https://nice.fr/FUZZUID?name=FUZZSTR&stamp=FUZZTME {"age": FUZZNUM, "date": "FUZZDTE" }
```

The format for each fuzz requests is basically _Verb + URL + Body_, and more precisely: 

```console
[GET|PUT|POST|DELETE|PATCH] URL [JSON Body]
```

Fuzz files can be written by hand by preferably they are generated automatically 
with correct fuzz types and embedded examples from Open API Spec (> 3.0).
